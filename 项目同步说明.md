# A2A 项目同步说明

日期：2026-06-01

当前远端 `main` 基线：`3d94a36 Add resident multi-workflow commander manager`

这份说明用于同步当前 A2A 项目的进展，并明确后续同学们要实现的 Agent 细节、接口边界、合并流程和验收标准。目标不是让每个人各自改一套，而是先统一公共协议，再并行实现各自负责的 Agent，最后稳定合并到主项目里。

## 一、当前项目进展

目前项目已经完成了几项关键基础能力，整体上不再只是一次性演示，而是可以动态选择 BPEL、并发推进多套流程、恢复、接管和继续执行的工作流系统。

### 1. 已完成的公共能力

- 工作流状态已经落盘，支持通过 `workflow_id` 恢复。
- Commander 支持 `resume`，可在新进程、新端口上继续接管同一个 workflow。
- `workflow_state_store.py` 已经负责 checkpoint 的保存、加载和删除。
- 附件协议已经统一为对象存储引用，不允许把二进制内容直接塞进任务消息。
- `attachment_uploader.py` 已支持把本地文件转成标准附件引用。
- Nacos 注册层已经加了心跳，默认 5 秒发送一次，并会过滤失联实例。
- Commander 已支持动态加载预先编写的 BPEL，可以按任务需求选择不同工作流。
- 单个 BPEL 内仍按业务依赖顺序推进；标记为 `dispatchMode="parallel"` 的 activity 会并发派发给多个同类型 Agent 实例。
- 已新增常驻 `CommanderWorkflowManager`，使用 Workflow 线程池同时推进多套 BPEL。
- 已支持 Agent 租约和资源锁：任务开始时 Agent 标记为 `busy`，结束或失败时恢复为 `idle`，避免多个 Workflow 抢占同一个实例。
- `--max-workflows` 可以限制同时运行的 Workflow 数量，超过限制的任务会进入 `queued` 状态。
- 每个 `workflow_id` 都拥有独立 checkpoint，保存到 `.a2a_state/workflows/<workflow_id>.json`。
- 已经补了进程重启恢复和 Commander failover/resume 的演示脚本。
- 已新增 `scripts/demo_workflow_manager.py`，可以一键展示租约、资源锁、多 Workflow 并发、排队和独立 checkpoint。
- 当前自动化回归测试共 `25` 项，恢复、附件、心跳、租约和多 Workflow 调度链路均已跑通。

### 2. 当前项目的工作方式

现在项目的主结构可以理解为三层：

- 常驻调度层：`CommanderWorkflowManager`、Workflow 线程池、并发上限、Agent 租约、资源锁。
- 公共控制面：单 Workflow Commander、BPEL 加载、checkpoint、恢复 API、附件协议、心跳、Nacos 服务发现。
- 业务实现层：Recon、Artillery、Evaluator、Assault 等具体 Agent 的实现细节。

你们后续协作时，公共控制面尽量由我统一维护，大家重点实现各自 Agent 的行为细节，最后再按统一接口接进来。

### 3. 当前并发模型

项目中有两种不同层次的并发：

| 并发层次 | 控制参数 | 作用 |
| --- | --- | --- |
| 多 Workflow 并发 | `--max-workflows` | 控制常驻 Manager 最多同时推进多少套 BPEL |
| 同类 Agent 并发 | `--max-workers` | 控制单个 BPEL activity 最多并发调用多少个同类型 Agent |

不同类型 Agent 之间仍遵循 BPEL 定义的前后依赖。例如 `recon -> artillery -> evaluator -> assault` 仍然按顺序推进。并发主要发生在两个位置：

- 多套互相独立的 BPEL 可以同时运行。
- 同一个 activity 可以并发调用多个同类型 Agent，例如多个 `artillery` 实例。

### 4. 当前租约边界

当前资源锁由单个常驻 `CommanderWorkflowManager` 统一维护，并把 `busy/idle` 状态同步到 Nacos metadata。它可以避免同一个 Manager 内的多个 Workflow 抢占同一个 Agent 实例。

如果后续部署多个 Manager 进程，需要进一步引入 Redis、数据库或 Nacos CAS 分布式锁，并补充 fencing token，避免跨进程抢占。

## 二、这次同步的核心目标

你们这轮任务的核心不是“重做一套项目”，而是：

1. 按统一协议实现各自负责的 Agent 逻辑。

也就是说，大家实现的不是孤立模块，而是“能被 Commander 调度的 Agent 能力单元”。

## 四、公共接口边界

这一部分最重要。大家开发前先对齐这些边界，不然后面很容易合并冲突。

### 1. 不要随便改的字段

以下字段建议作为公共协议固定下来：

- `workflow_id`：整条流程的唯一标识。
- `work_item`：单个工作项的唯一标识。
- `attachments`：附件引用列表，只允许对象存储引用。
- `checkpoint`：工作流状态快照。
- `role`：Agent 角色名，例如 `recon`、`artillery`、`evaluator`、`assault`。
- `status`：任务/工作流状态，例如 `Accepted`、`Working`、`Completed`、`paused`、`completed`。
- `workflow_status`：Manager 中的流程状态，例如 `queued`、`running`、`paused`、`completed`、`failed`。
- `activatity`：当前项目 checkpoint 中使用的活动字段命名，例如 `current_activatity`、`workflow_activatity`。

### 2. Agent 最少要遵守的接口行为

每个 Agent 至少需要满足这四点：

- 能被 `/.well-known/agent-card` 发现。
- 能通过鉴权。
- 能处理 `sendMessage`。
- 如果是长任务，能处理 `sendMessageStream` 或等价的流式反馈。

### 3. 返回值建议保持统一

建议同学们返回值尽量采用类似下面这种结构：

```json
{
  "work_item": "workflow-xxx:activatity-005-suppressbeachsector",
  "status": "Accepted",
  "role": "artillery",
  "message": "Artillery completed command=suppress_beach_sector_A"
}
```

这样 Commander 在接收结果时比较容易统一处理，也便于后续做幂等和恢复。

## 五、开发要求

### 1. 每个人先明确自己的 Agent 职责

不要一上来就改公共逻辑，先把自己负责的 Agent 具体要做什么写清楚：

- 输入是什么。
- 输出是什么。
- 成功时返回什么。
- 失败时返回什么。
- 流式任务要输出哪些阶段。

### 2. 保持幂等

任务恢复或重复提交时，不能重复产生副作用。

建议大家在自己负责的逻辑里都遵守：

- 同一个 `work_item` 只能执行一次业务副作用。
- 如果重复收到同一个 `work_item`，优先返回缓存结果。
- 流式结果也尽量能重放或缓存。

### 3. 不要把临时调试代码提交进主流程

开发时可以调试，但最终合并时不要留这些东西：

- 临时 print。
- 写死的 IP / 端口。
- 只对你本机有效的 mock 逻辑。
- 只用于实验的分支判断。

如果要保留 mock，建议改成显式参数，例如 `--mock-eval-score`、`--mock-decision` 这种形式。

### 4. 每个 Agent 都要补测试

至少要有一条能证明“这个 Agent 可以被接入”的测试或演示。

建议测试点包括：

- 能否被发现。
- 能否被鉴权后调用。
- `sendMessage` 是否返回正确结构。
- 流式任务是否能返回阶段性事件。
- 重复调用同一 `work_item` 时是否保持幂等。

## 六、建议的开发流程

### 第一步：先开同步会

先花 15 到 30 分钟把下面几件事对齐：

- 现在项目已经做到了什么。
- 你们各自负责哪一个 Agent。
- 公共字段和返回结构怎么定。
- 合并时谁来做最终接入。
- 验收标准是什么。

### 第二步：统一分支规则

建议所有人都从同一个稳定基线切分支，分支名尽量统一：

- `feature/<name>-recon`
- `feature/<name>-artillery`
- `feature/<name>-evaluator`
- `feature/<name>-assault`

如果是修 bug：

- `fix/<name>-heartbeat`
- `fix/<name>-resume`

### 第三步：每人只改自己负责的模块

尽量不要跨多个 Agent 同时改，也不要多人同时改同一个公共协议文件。

推荐原则是：

- Agent 逻辑改各自目录。
- 公共协议改动先提出来讨论。
- 需要改 Commander 的地方先和我确认。

### 第四步：先本地跑通，再提交合并

每个同学完成后，先在本地至少跑通一次：

- 对应 Agent 的最小示例。
- 对应的单元测试。
- 如果涉及工作流，还要跑恢复 / resume 演示。

### 第五步：统一合并

合并时按照下面顺序：

1. 先把各自分支同步到最新主分支。
2. 再解决冲突。
3. 最后统一跑全量测试。
4. 全部通过后再合并到主线。

## 七、合并时的验收标准

每个同学的 PR 或合并请求，至少要满足以下条件：

- 代码能跑。
- Agent 能被发现。
- Agent 能正常接收任务。
- 返回结果结构符合公共协议。
- 如果有流式任务，流式输出正常。
- 如果有状态保存，恢复后能继续工作。
- 新增逻辑不破坏已有测试。

如果是改核心控制面，还要额外满足：

- 不影响 checkpoint 结构。
- 不影响 workflow_id 恢复。
- 不影响心跳过滤和 failover。
- 不允许多个 Workflow 同时领取同一个 Agent 实例。
- Agent 完成或失败后必须释放租约并恢复为 `idle`。
- 超过 Workflow 并发上限时，新任务必须进入 `queued`，不能绕过线程池。
- 每个 `workflow_id` 的 checkpoint 必须相互隔离。

## 八、常驻 Manager 运行与验证

### 1. 启动 Manager

本地模拟模式不依赖 Nacos，适合先验证 BPEL 调度：

```bash
cd /home/yl/yl/jzz/A2A
./venv/bin/python -u commander_agent/main.py \
  --mode local \
  --serve-workflow-manager \
  --manager-port 8021 \
  --max-workflows 2
```

远端模式会使用 Nacos 和真实 Agent：

```bash
./venv/bin/python -u commander_agent/main.py \
  --mode remote \
  --serve-workflow-manager \
  --manager-port 8021 \
  --max-workflows 4
```

### 2. 查询接口

```bash
curl http://127.0.0.1:8021/health
curl http://127.0.0.1:8021/workflows
curl http://127.0.0.1:8021/leases
curl 'http://127.0.0.1:8021/workflows/<workflow_id>?checkpoint=true'
```

### 3. 一键演示

```bash
./venv/bin/python -u scripts/demo_workflow_manager.py
```

该脚本会自动验证：

1. Agent 状态从 `idle -> busy -> idle`。
2. 一个 Agent 被租用时，其他 Workflow 无法重复抢占。
3. Agent 释放后，其他 Workflow 可以重新领取。
4. 同时提交三套 BPEL，在 `--max-workflows 2` 时前两套并发运行，第三套进入 `queued`。
5. 三套 BPEL 分别生成独立 checkpoint。

验证通过时会输出：

```text
[PASS] queued state, concurrent execution, and independent checkpoints verified.
```
