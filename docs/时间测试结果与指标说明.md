# A2A 阶段耗时测试结果与指标说明

本文档记录 A2A 多 Agent 项目的阶段耗时测试结果，并解释各个测试指标的含义。

## 1. 测试背景

本次测试用于观察 A2A 调用链路中各阶段的耗时，包括：

- 通过 Nacos 发现目标 Agent 的路由耗时。
- 获取目标 Agent Card 的耗时。
- 模拟鉴权耗时。
- 任务提交响应耗时。
- Agent 流式任务完成耗时。
- 单阶段端到端总耗时。

测试脚本：

```bash
/home/yl/yl/jzz/A2A/venv/bin/python timing_probe.py \
  --service A2A-Timing-Run2 \
  --roles recon,artillery,evaluator,assault \
  --iterations 3
```

说明：

- `--iterations 3` 表示每个角色连续测试 3 轮。
- `--roles recon,artillery,evaluator,assault` 表示测试 4 类 Agent。
- `--service A2A-Timing-Run2` 是本次测试在 Nacos 中注册的测试服务名，指向当前运行中的 4 个 Agent。
- 结果单位均为毫秒，记作 `ms`。

## 5. 指标含义

### 5.1 `route_discovery`

通过 Nacos 查找目标 Agent 的耗时，也就是路由时间。

例如需要调用炮兵 Agent 时，会按如下条件查询：

```text
serviceName = A2A-Timing-Run2
metadata.role = artillery
metadata.status = idle
```

从发起 Nacos 查询到拿到可用实例的 IP 和端口，这段时间就是 `route_discovery`。

该指标可以用于评估服务发现与动态路由的性能。

### 5.2 `agent_card_discovery`

获取目标 Agent Card 的耗时。

对应请求：

```text
GET /.well-known/agent-card
```

Agent Card 中包含目标 Agent 的名称、角色、能力描述、鉴权方式和消息接口地址。

该指标可以用于评估 A2A Discovery 阶段的网络与服务响应开销。

### 5.3 `authentication`

模拟鉴权耗时。

当前项目并未实现真实 OAuth2 或 JWT 签发，而是访问 mock auth server：

```text
http://127.0.0.1:8080/post
```

随后客户端生成模拟 token：

```text
mock-jwt-token-abcd
```

所以该指标代表当前 mock 鉴权链路的耗时，不代表真实生产鉴权系统的耗时。

### 5.4 `task_submit_ack`

调用同步任务接口并收到接收响应的耗时。

对应请求：

```text
POST /sendMessage
```

该阶段只表示 Agent 已经接收任务，并返回类似 `Accepted` 的响应。

它不表示任务真正执行完成。

### 5.5 `task_time_to_first_event`

调用流式任务接口后，收到第一个 SSE 事件的耗时。

对应请求：

```text
POST /sendMessageStream
```

该指标可以理解为 Agent 开始反馈任务进度的延迟。

如果该指标较高，说明 Agent 接收流式任务后到首次反馈之间存在较大等待。

### 5.6 `task_stream_completion`

流式任务从开始到结束的耗时。

该指标更接近“Agent 完成一个任务需要多长时间”。

当前实现中：

- `recon`、`evaluator`、`assault` 使用 `A2ABaseAgent` 的默认流式实现，内部有一次 `asyncio.sleep(1)`，因此任务完成时间约为 `1000 ms`。
- `artillery` 重写了流式实现，内部有三次 `asyncio.sleep(1)`，因此任务完成时间约为 `3000 ms`。

所以本次测试中，炮兵 Agent 的任务完成耗时明显高于其他 Agent，这是符合代码实现预期的。

### 5.7 `phase_total`

单个 Agent 阶段的端到端总耗时。

它包含：

```text
route_discovery
+ agent_card_discovery
+ authentication
+ task_submit_ack
+ task_stream_completion
+ 少量 Python 执行与统计开销
```

例如 `artillery` 的 `phase_total` 约为 `3019 ms`，其中绝大部分来自 `task_stream_completion` 的 `3005 ms`。

该指标适合用来评估一个完整 Agent 阶段的总成本。

### 5.8 `total_elapsed_ms`

整个测试脚本的总耗时。

本次测试执行了：

```text
recon -> artillery -> evaluator -> assault
```

并重复 3 轮，因此 `total_elapsed_ms` 是这 3 轮、4 个角色串行执行后的总耗时。

## 6. 结果解读

从本次测试结果看：

- Nacos 路由发现耗时整体较低，平均约 `4 ms`。
- Agent Card 获取耗时稳定，平均约 `3 ms`。
- mock 鉴权耗时稳定，平均约 `1.8 ms`。
- 同步任务提交响应大多在 `3 ms` 左右。
- 默认 Agent 的任务完成时间约 `1 秒`，主要由代码中的 `asyncio.sleep(1)` 决定。
- 炮兵 Agent 的任务完成时间约 `3 秒`，主要由代码中的三次 `asyncio.sleep(1)` 决定。
- 当前总耗时主要由 Agent 自身模拟任务执行时间决定，而不是路由、发现或鉴权开销。

## 7. 测试注意事项

本次测试过程中发现两个会影响结果的环境问题，已在代码中处理：

- 当前环境存在 `HTTP_PROXY` / `HTTPS_PROXY`，Python `requests` 默认会使用代理访问 `127.0.0.1`，导致本地 Agent 或 Nacos 调用异常。
- 当前 `nacos-sdk-python<2.0.0` 访问 Nacos 2.3.2 时会出现 `All server are not available`，因此在 `NacosRegistry` 中加入了 HTTP API fallback。

因此后续测试建议继续使用当前版本的 `timing_probe.py` 与修正后的 Nacos/A2AClient 调用逻辑。

### 8.2 与原版路由对比

为了只比较“路由决策”本身，下表将原版的 `route_discovery` 与大模型路由的 `llm_route_decision + route_discovery` 对比。

| 阶段 | 原版 Nacos 路由均值 | 大模型决策耗时 | 大模型后 Nacos 路由 | 大模型总路由耗时 | 增加耗时 | 约为原版倍数 |
|---|---:|---:|---:|---:|---:|---:|
| `recon` | 4.350 | 2136.189 | 5.257 | 2141.446 | 2137.096 | 492.3x |
| `artillery` | 4.658 | 1031.131 | 6.193 | 1037.324 | 1032.666 | 222.7x |

### 8.3 对比结论

从本次测试看：

- 原版路由主要是 Nacos 查询，平均只有 `4 ms` 左右。
- 大模型路由新增了一次模型调用，单次决策耗时约 `1.0 s` 到 `2.1 s`。
- 大模型完成角色判断之后，后续 Nacos 查询仍然保持在 `4 ms` 到 `6 ms` 左右。
- 因此，大模型路由的主要成本来自 `llm_route_decision`，不是 Nacos，也不是 A2A Agent 调用。
- 如果任务类型明确、规则固定，原版基于 metadata 的路由更快。
- 如果任务描述复杂、需要语义理解、需要动态判断执行者，则可以接受更高耗时使用大模型路由。
