# A2A 小组代码合并规范

## 1. 目标

这份规范的目标，不是把所有人的代码都强行合成一份“最终版”，而是让不同成员在同一个 A2A 项目基础上协作时，能够做到：

- 接口稳定，别人容易接；
- 模块边界清楚，冲突少；
- 状态和配置统一，方便联调；
- 出问题能回滚、能恢复、能定位；
- 合并前后都能通过统一测试。

对于这个项目来说，最适合作为团队合并基础的部分，应该是“稳定控制面 + 稳定协议层 + 可插拔 Agent 实现”。也就是说，团队成员尽量不要直接改 Commander 主流程的核心约定，而是围绕固定协议去扩展自己的 Agent、任务逻辑或业务能力。

## 2. 推荐的合并原则

### 2.1 先定接口，再写实现

所有人先对齐以下内容，再开始写功能：

- `workflow_id` 如何生成、如何传递、如何恢复；
- `task_id` 如何构成，是否需要幂等；
- `attachments` 只传什么格式，是否允许内联内容；
- Agent 的输入输出字段是否统一；
- 失败时返回什么状态，是否支持重试；
- 哪些字段属于公共协议，哪些字段只属于本地模块。

原则上，协议先稳定，业务实现后补。这样团队成员各自实现时，不会因为字段名不同、数据结构不同而频繁冲突。

### 2.2 控制面和业务面分离

建议把项目拆成两层看：

- 控制面：Commander、恢复 API、checkpoint、心跳、接管逻辑；
- 业务面：Recon、Artillery、Evaluator、Assault 等具体 Agent。

团队协作时，控制面尽量由少数人维护，业务面允许多人并行扩展。这样可以避免多人同时改 Commander 主流程，导致冲突密集、行为不一致。

### 2.3 公共协议文件单独维护

像下面这些文件，建议只由少数人维护，或者通过评审后再改：

- `workflow_payloads.py`
- `workflow_state_store.py`
- `commander_agent/recovery_api.py`
- `registry/nacos_manager.py`
- `a2a_protocol/server.py`

这些文件相当于“项目公共契约”。一旦改动，所有 Agent 和脚本都可能受影响，所以不要让每个人随意改自己的分支逻辑。

### 2.4 业务逻辑尽量下沉到独立模块

如果团队里有人要新增功能，优先新增独立模块，而不是往 Commander 里堆分支。

例如：

- 新 Agent -> 新建 `xxx_agent/`
- 新的任务解析 -> 新建 `commander_agent/xxx_planner.py`
- 新的附件处理 -> 新建 `attachment_xxx.py`
- 新的恢复检查 -> 新建 `recovery_xxx.py`

这样做的好处是：

- 冲突更少；
- 代码更容易 review；
- 出问题更容易单独回滚；
- 其他人合并时不需要理解所有细节。

## 3. 最适合团队共用的“基础分支”做法

### 3.1 保留一个稳定主分支

建议把当前这个项目作为共同基础分支，主分支只放已经验证过的稳定能力：

- workflow 持久化；
- 恢复 API；
- 附件协议；
- 心跳和 failover；
- 已经通过测试的脚本和文档。

所有新需求都从这个稳定基线分出去开发，不要在主分支上边试边改。

### 3.2 每个人从基础分支切自己的功能分支

推荐命名方式：

- `feature/<name>-<topic>`
- `fix/<name>-<bug>`
- `experiment/<name>-<idea>`

例如：

- `feature/zhangsan-evaluator`
- `fix/lisi-heartbeat`
- `experiment/wangwu-routing`

这样一眼就能看出是谁在做什么，后续合并和回滚也方便。

### 3.3 合并时先做 rebase 或同步基线

每次合并前，先把自己的分支同步到当前基础分支，再发起合并请求。不要长期落后主分支很多提交，否则容易出现：

- 接口字段不一致；
- 同一文件多人改动导致冲突爆炸；
- 旧逻辑覆盖新 checkpoint 结构；
- 本地能跑，合并后不能跑。

## 4. 让别人容易合并的关键做法

### 4.1 固定输入输出

每个模块都要写清楚输入输出。

建议统一格式：

- 输入：JSON / Python dict / 统一附件结构；
- 输出：标准状态字段 + 结果字段 + 错误字段；
- 错误：统一使用可读的错误码或错误文本。

例如，Agent 的返回尽量保持这种风格：

```json
{
  "task_id": "workflow-xxx:2:artillery",
  "status": "Accepted",
  "role": "artillery",
  "message": "Local artillery completed command=suppress_beach_sector_A"
}
```

这样别人接入时，不需要猜返回格式。

### 4.2 降低全局共享状态

最容易引发合并冲突的地方，往往是全局状态。建议：

- 工作流上下文统一走 checkpoint；
- 不要在多个地方同时写同一份状态；
- 共享配置集中到环境变量或配置文件；
- 运行时临时变量不要散落在多个模块里。

这会让不同人的改动更独立，合并冲突自然会少很多。

### 4.3 不要把临时调试逻辑提交进核心路径

团队开发时，最常见的问题是把临时 print、临时 mock、临时开关一直留在主流程里。建议：

- 调试脚本放 `scripts/`；
- 临时实验放独立分支；
- 真正要保留的实验能力，改成明确参数，例如 `--mock-eval-score`、`--resume`；
- 不要把某个人的本地路径、IP、端口写死到公共代码里。

### 4.4 为新增模块配测试

谁加功能，谁至少补一条测试。

建议重点补这些类型的测试：

- checkpoint round trip；
- resume 是否从正确步骤继续；
- 附件协议是否拒绝非法数据；
- 心跳失联时是否过滤掉 stale instance；
- 新 Agent 是否能被正常发现并调用。

团队合并时，测试比口头说明更有说服力。

## 5. 推荐的团队协作流程

### 5.1 开发前先对齐边界

在开始写代码前，先确定：

- 这个功能改控制面，还是改业务面；
- 是否会影响公共协议；
- 是否会影响 checkpoint 结构；
- 是否会影响 Agent 注册和发现；
- 是否需要新增脚本或测试。

### 5.2 开发中保持小步提交

不要一次提交太多不相关的改动。建议一个 PR 只做一类事情，例如：

- 只改恢复逻辑；
- 只改附件逻辑；
- 只加一个 Agent；
- 只补一个脚本；
- 只写一个测试集合。

小 PR 更容易 review，也更容易合并。

### 5.3 合并前做统一检查

建议所有人至少跑下面这些检查：

- 单元测试；
- 启动脚本或 demo 脚本；
- 关键文档是否同步更新；
- 新字段是否和旧 checkpoint 兼容。

如果涉及核心协议，最好还要做一次完整联调。

## 6. 针对这个项目的具体建议

结合当前 A2A 的现状，最适合作为“小组代码合并基础”的是下面这套稳定层：

- `commander_agent/main.py`：只保留流程控制和状态推进；
- `workflow_state_store.py`：统一 checkpoint 落盘；
- `commander_agent/recovery_api.py`：统一恢复入口；
- `workflow_payloads.py`：统一附件协议；
- `registry/nacos_manager.py`：统一服务发现和心跳；
- `local_runtime.py`：统一本地模拟与幂等验证。

团队成员如果要加东西，最好围绕这些稳定层向外扩展，不要直接改它们的公共语义。

更具体地说：

- 新业务能力，优先做成新 Agent；
- 新流程分支，优先做成 Commander 的独立规划函数；
- 新数据类型，优先做成独立协议字段或附件 meta；
- 新恢复策略，优先在 recovery API 外围增加，不直接破坏旧接口。

## 7. 建议的最终落地形式

如果你要把这个项目正式作为小组合并基础，建议再补三样东西：

1. 一份统一的协作规范，也就是这份文档；
2. 一套固定的接口说明，说明 workflow、task、attachments、checkpoint 的格式；
3. 一套统一的验证命令，要求大家合并前都跑同样的测试和 demo。

这样做之后，别人接入你的项目时，最少会遇到三件确定的事：

- 该改哪里；
- 该遵守什么格式；
- 怎么证明自己改对了。

这就是让团队代码更容易合并的关键。
