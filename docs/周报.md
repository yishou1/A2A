# A2A 项目周报

## 一、本周主要工作

本周围绕 A2A 多智能体协同项目进行了路由机制分析、耗时测试、大模型路由扩展和结果文档整理。

首先，对项目当前路由机制进行了梳理。原版系统采用 Nacos metadata 路由方式，由代码提前确定目标 Agent 的 `role`，例如 `recon`、`artillery`、`evaluator`、`assault`，再通过 Nacos 查询健康实例并调用对应 Agent。该方式不依赖大模型，路由耗时稳定在毫秒级。

其次，实现了阶段耗时测试脚本 `timing_probe.py`，用于测量 A2A 调用链路中的关键耗时，包括 Nacos 路由发现、Agent Card 获取、模拟鉴权、任务提交响应、流式任务完成时间和单阶段总耗时。测试结果已整理到 Markdown 文档中。

此外，在原有 Nacos 路由基础上增加了大模型路由测试能力。新增 `--llm-route` 参数后，系统会先调用大模型判断目标角色，再使用 Nacos 查询对应 Agent。该机制验证了“大模型语义判断 + Nacos 服务发现”的可行性。

## 二、测试结果概况

原版 Nacos 路由测试结果较稳定：

| 阶段 | 原版 Nacos 路由均值 |
|---|---:|
| recon | 4.350 ms |
| artillery | 4.658 ms |
| evaluator | 4.034 ms |
| assault | 3.991 ms |

可以看出，单纯依赖 Nacos metadata 查询时，路由基本在 `4-6 ms` 左右，满足低延迟要求。

大模型路由方面，测试了多个模型。其中 `gpt-4.1-mini-free` 曾成功完成一轮四角色路由，耗时约 `1.0-2.1 s`；`Qwen/Qwen3.5-4B` 后续也成功完成路由，但耗时明显更长，约 `6.5-14.7 s`。

| 模型 | 路由判断耗时表现 | 结论 |
|---|---:|---|
| 原版 Nacos | 约 4-6 ms | 最快，适合固定规则路由 |
| gpt-4.1-mini-free | 约 1.0-2.1 s | 大模型方案中相对较快 |
| Qwen/Qwen3.5-4B | 约 6.5-14.7 s | 可正确路由，但延迟较高 |

同时也遇到了一些 API 层问题，包括模型无权限、免费额度限制、API Key 无效等，相关失败测试也已记录到文档中，避免误将失败响应耗时当成有效路由耗时。

## 三、关键结论

从本周测试结果看，如果当前目标是“时间短、响应快”，原版 Nacos metadata 路由明显优于大模型路由。Nacos 路由只需要毫秒级，而大模型路由会额外增加一次远程模型调用，通常带来秒级延迟。

大模型路由的主要耗时来自 `llm_route_decision`，而不是 Nacos 查询，也不是 Agent 调用。即使加入大模型后，后续 Nacos 查询仍然稳定在几毫秒。

因此，大模型更适合作为复杂自然语言任务的语义兜底，而不适合作为所有任务的默认实时路由方式。

## 四、优化建议

后续建议采用混合路由策略：

```text
固定任务 / 明确 command -> 直接 Nacos 路由
语义模糊 / 无法规则判断 -> 调用大模型判断 role
```

也可以进一步引入轻量语义路由方式，例如关键词规则、Agent 能力描述匹配或 embedding 相似度检索。这样可以在不依赖大模型的情况下获得一定语义能力，并保持低延迟。

推荐路线：

```text
第一阶段：关键词规则路由
第二阶段：embedding 相似度路由
第三阶段：大模型兜底
```
