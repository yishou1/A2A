# 5-29 荆振洲 周报

## 一、本周工作概述

本周围绕 A2A 多智能体协同项目，重点推进了“可恢复、可接管、可观测、可演示”四个方向的能力建设。相比之前只关注单次任务执行，本周的工作重点已经转向工作流级别的可靠性设计：当 Commander 宕机、进程重启、端口切换或者某个 Agent 临时不可用时，系统仍然要能够继续执行，并且尽量避免重复执行、状态丢失和错误接管。

从结果上看，本周已经把几个关键链路串起来了：

- 工作流状态可以落盘保存，并通过 `workflow_id` 恢复；
- 恢复接口可以在新进程、新端口上继续接管同一个工作流；
- 附件从“内联传输”统一收敛为“对象存储引用”；
- Agent 注册层加入了 5 秒心跳与失联过滤；
- 已经补齐了可运行的演示脚本和回归测试。

整体上，这一周的工作不只是“加功能”，更重要的是把 A2A 从一个偏演示型的单进程流程，推进成一个更接近真实生产控制面的工作流系统。

## 二、详细工作内容

### 1. 工作流状态持久化与断点续跑

本周首先解决的是 Commander “只存在于内存里”的问题。原先工作流上下文如果只保存在进程内存中，那么一旦进程退出、容器重启或者机器异常，正在执行的流程就会直接中断，后续也无法恢复。

为了解决这个问题，新增了工作流状态仓库 `WorkflowStateStore`，把每个工作流的状态保存到 `.a2a_state/workflows/<workflow_id>.json`。这样一来，工作流就不再依赖单次进程生命周期，而是变成了可持久化、可重启、可恢复的状态对象。

这一层的关键意义有三点：

- `workflow_id` 变成了整个工作流的稳定锚点，恢复时只需要这个 ID 就能找到对应 checkpoint；
- checkpoint 不再是临时变量，而是可以落盘的“断点”；
- 恢复时不会从头再跑，而是从最后一次保存的步骤继续往下执行。

同时，Commander 的动态工作流逻辑也同步调整为支持 `resume` 模式。恢复后，系统会根据 checkpoint 中保存的 `workflow_step` 继续推进，而不是重新初始化上下文。这样可以保证宕机之前已经完成的步骤不会被重跑，降低重复执行的风险。

### 2. 恢复 API 与外部接管

为了让“恢复”不局限于同一个进程内，还新增了恢复 API，核心入口在 `commander_agent/recovery_api.py`。这个接口提供了几个关键能力：

- `GET /health`：用于判断控制面是否存活；
- `GET /workflows/{workflow_id}`：用于查看某个工作流当前保存的状态；
- `POST /workflows/{workflow_id}/resume`：用于从 checkpoint 恢复并继续执行；
- `POST /workflows/{workflow_id}/takeover`：用于显式接管已有流程。

这部分工作的价值在于，恢复动作被从 Commander 主执行流程中拆出来了。也就是说，哪怕原来的 Commander 已经挂了，只要 checkpoint 还在，新的进程就可以在另外一个端口上重新启动，并继续接手同一个工作流。

从架构上看，这一改动让“流程执行”和“控制面接管”分离得更清楚了：

- 执行逻辑负责推进业务步骤；
- 恢复 API 负责提供外部接管入口；
- checkpoint 负责把两者连接起来。

这也是本周最关键的一项能力，因为它直接决定了系统是否真正具备宕机后的续跑能力。

### 3. 附件协议与上传链路规范化

本周还把复杂参数传递中的附件协议正式收敛了。之前如果把图片、视频、文件等内容直接内联进任务消息，不仅消息体会变大，而且恢复、重试和转发都会变得很麻烦。为此，附件现在统一采用对象存储引用的方式传递，而不再直接携带二进制内容。

目前附件的核心思路是：

- 任务消息里只保留引用，不保留原始大文件；
- 附件必须包含 `uri` 和 `checksum`；
- 可附带 `kind`、`mime_type`、`size_bytes`、`name` 等元信息；
- 如果是图片、视频等媒体，还可以携带宽高、时长等结构化信息；
- 所有附件都会经过标准化处理，保证恢复后仍然能正确识别。

同时新增了 `attachment_uploader.py`，用于把本地文件上传成标准附件引用。这个上传助手会先计算文件摘要，再推断 MIME 类型和附件类型，最后返回符合协议的 attachment ref。这样做的好处是：

- 上传和引用解耦；
- 大文件不再塞进任务消息；
- 附件可以在恢复后继续稳定读取；
- 后续可以无缝对接 HTTP PUT、预签名地址或对象存储后端。

这一块的工作让 A2A 的参数传递从“能用”升级到了“适合恢复、适合重试、适合长期维护”。

### 4. 心跳机制与宕机安全

为了处理 Agent 或 Commander 宕机后的发现问题，本周又补了心跳机制。现在 Agent 注册到 Nacos 时，会自动启动心跳线程，默认每 5 秒上报一次状态；发现服务时，则会根据心跳时间过滤掉过期实例，避免拿到已经失联的节点。

这部分的核心改动有两层：

- 注册侧：加入 `heartbeat_interval`，让 Agent 在注册后持续发送心跳；
- 发现侧：通过 `heartbeat_ts` 和宽限时间过滤 stale instance，避免“注册还在、服务已死”的情况。

这意味着服务发现不再只看 Nacos 里有没有实例，而是结合最近一次心跳是否新鲜来判断实例是否可用。对于宕机安全来说，这很重要，因为“还在注册中心”不等于“还能正常服务”。

在 Commander 侧，还补了一个 failover 演示流程：

1. 先跑一个可恢复 checkpoint；
2. 启动主 Commander API；
3. 按固定间隔做健康检查；
4. 模拟主 Commander 宕机；
5. 检测到连续失联后，在新端口启动备用 Commander；
6. 通过 `resume` 接口接管同一个 workflow；
7. 从 checkpoint 的中间态继续执行，直到完成。

这一整套流程说明系统已经具备“检测失联 -> 新实例接管 -> 继续执行”的基本闭环。

### 5. 演示脚本与回归测试

为了把以上能力变成可验证、可复现的结果，本周还补了两个重要脚本：

- `scripts/demo_resume_after_restart.py`
- `scripts/demo_commander_failover_resume.py`

前者用于演示“同一个 workflow 在进程重启后继续执行”；后者用于演示“主 Commander 宕机后，新端口上的 Commander 接管并 resume”。这两个脚本把原本抽象的恢复能力变成了可以直接运行的闭环演示。

同时，测试也已经补齐并验证通过，重点覆盖了：

- 工作流状态存取与 checkpoint round trip；
- Commander 的 resume 行为；
- 本地运行时的幂等 replay；
- 附件协议校验与上传助手；
- 恢复 API 的接管能力；
- Agent 心跳与过期实例过滤。

目前项目测试已经通过，说明这一轮改动没有把已有主流程打坏，恢复链路和心跳链路也都能稳定运行。

### 6. 关键术语梳理

为了方便后续沟通，这里也把几个核心概念重新梳理一下：

- `workflow_id`：整条工作流的唯一标识，是恢复、查询、接管的主键；
- `task_id`：单个任务步骤的唯一标识，主要用于幂等与任务级追踪；
- `checkpoint`：工作流在某个时刻的持久化状态，可以用于恢复；
- `attachments`：工作流中携带的外部资源引用，当前统一要求用对象存储 URI 表达；
- `uploader`：把本地文件转换成标准附件引用的上传助手；
- `local_runtime.py`：本地模拟运行时，用来在不依赖外部服务的情况下验证流程和幂等行为。

这些概念加在一起，实际上构成了 A2A 当前的“可恢复执行模型”：流程用 `workflow_id` 锚定，任务用 `task_id` 防重，资源用 `attachments` 引用，状态用 `checkpoint` 落盘，文件由 `uploader` 转换，本地则可以通过 `local_runtime.py` 进行离线模拟。

## 四、验证结果

本周的几项关键验证都已经通过：

- 全量测试通过，说明恢复、附件、心跳相关逻辑没有引入明显回归；
- `scripts/demo_resume_after_restart.py --reset` 可以成功演示进程重启后的恢复；
- `scripts/demo_commander_failover_resume.py --reset` 可以成功演示主 Commander 宕机、换端口接管、恢复执行的完整闭环；
- 附件示例也已经可以正常上传并在 checkpoint 中保持引用。

这说明目前的实现不仅停留在设计层面，而是已经具备实际运行和演示条件。

## 五、存在的问题与后续优化方向

虽然本周已经把“可恢复”这件事做出来了，但从宕机安全角度看，仍然还有一些进一步增强的空间：

- 需要进一步防止 split-brain，避免旧 Commander 和新 Commander 同时推进同一个 workflow；
- 需要给恢复接口补上更明确的鉴权和授权控制；
- checkpoint 还可以进一步增加版本号、校验和和更强的落盘保障；
- 对于外部副作用较强的任务，还需要更严格的幂等与补偿机制；
- 未来可以把当前 demo watcher 进一步提升为独立控制器，而不是只放在脚本里。

如果后续继续推进，优先级建议是：先做 workflow 级锁和 fencing token，再补恢复接口鉴权，最后再强化 checkpoint 的一致性保障。

## 六、总结

本周工作的核心成果，是把 A2A 从“单次执行、依赖内存”的流程，推进到“可落盘、可恢复、可接管、可演示”的工作流系统。工作流恢复、附件规范、心跳检测和 failover 演示这几条线已经基本闭合，说明系统在面对宕机、重启和接管时，已经具备了初步的工程化能力。

后续如果继续围绕生产级可靠性推进，重点就不再只是“能恢复”，而是“恢复得是否安全、是否唯一、是否不会重复推进”。这也是下一阶段最值得继续打磨的方向。
