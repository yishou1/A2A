# A2A 项目周报（2026.05.22）

## 一、本周工作概述

本周围绕 A2A 多智能体协同项目，重点完成了工作流恢复能力、复杂参数传递规范、外部接管接口以及相关回归测试的补充工作。同时，对项目现有的路由、通信与本地模拟机制进行了进一步梳理，明确了后续可持续扩展的技术方向。

## 二、重点工作进展

### 1. 工作流状态持久化与断点续跑

针对 Commander 当前以内存字典维护上下文、进程重启后无法接续任务的问题，补充了持久化工作流状态存储能力。新增的工作流状态仓库支持按 workflow_id 保存、加载和删除 checkpoint，使工作流状态能够脱离单进程内存独立存在。

在此基础上，Commander 的动态工作流流程已改造为可恢复执行模式：

- 支持通过 workflow_id 恢复历史流程上下文；
- 支持在每个步骤前后落盘 checkpoint；
- 支持任务失败后的暂停与重试；
- 支持在恢复后继续执行尚未完成的步骤，而不是从头重跑。

### 2. 外部接管恢复接口

为满足“新 command 直接接续旧任务”的需求，新增了恢复 API。该接口可根据 workflow_id 读取持久化状态，并由外部进程接管继续执行。

恢复接口的作用主要体现在以下几个方面：

- 能够显式查询指定 workflow_id 的当前状态；
- 能够在外部进程中恢复指定工作流；
- 能够将恢复动作与工作流执行逻辑解耦；
- 便于后续接入独立的调度器、控制台或运维入口。

### 3. 复杂参数传递与附件协议规范

针对图片、视频等复杂参数，正式整理并落地了附件协议规范。新的约定要求附件仅以对象存储引用形式传递，不再直接携带二进制内容或内联文本。

当前附件协议包含以下核心信息：

- 资源 URI；
- 校验信息，例如 sha256；
- 媒体类型，例如 image 或 video；
- mime_type、size_bytes、name 等基础属性；
- 宽高、时长等媒体元信息；
- 可扩展 meta 字段。

同时新增上传助手，支持将本地文件上传到 HTTP 预签名地址或自定义对象存储后端，再生成符合协议的附件引用。这样可以把大文件传输从任务消息中剥离出来，降低链路复杂度，并提高恢复和重试的一致性。

### 4. 回归测试补充

围绕新增的恢复能力与附件协议，补充了对应单元测试，覆盖以下场景：

- 工作流状态的保存与读取；
- Commander 基于 workflow_id 的恢复；
- 本地运行时重复提交的幂等行为；
- 附件协议对对象存储引用的校验；
- 恢复接口按 workflow_id 接管并继续执行；
- 本地文件上传到对象存储并生成规范化附件引用。

## 三、验证结果

本周新增能力已完成本地验证。当前项目测试用例通过，说明恢复链路、附件协议与上传助手在当前实现下运行正常，未引入明显回归问题。

## 四、主要结论

本周工作的核心成果是将 A2A 项目从“单次执行、依赖内存状态”的演示型流程，推进到“可持久化、可恢复、可接管”的工作流形态。与此同时，附件传递方式也从内联内容升级为对象存储引用模式，更适合图片、视频和其他大体积参数的协同传递。

从架构角度看，本周的改动进一步明确了三层职责：

- Workflow 负责流程推进；
- State Store 负责状态持久化；
- Attachments 负责外部资源引用与校验。
